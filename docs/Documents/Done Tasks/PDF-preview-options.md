# Превью PDF в админке (Agreement, Floorplans)

Нужна мини-версия документа, чтобы до открытия видеть, что внутри. Ниже варианты — выберите, какой реализовать.

---

## Контекст

- **Agreement** и **Floorplans** принимают: `.pdf`, `.doc`, `.docx`.
- Для **PDF** можно показать реальное превью (первая страница).
- Для **.doc/.docx** браузер сам не рендерит — только иконка + имя файла или конвертация на бэкенде.

---

## Вариант 1 — PDF.js (Mozilla)

**Идея:** Рисуем первую страницу PDF на canvas и показываем как миниатюру.

- **Плюсы:** Надёжно, работает с любым URL (в т.ч. Vercel Blob), единый вид во всех браузерах.
- **Минусы:** Нужна библиотека `pdfjs-dist` (~500 KB), чуть сложнее код.
- **Реализация:** Компонент загружает PDF по `fileUrl`, рендерит 1-ю страницу в маленький canvas (например 200×280 px), по клику — открыть полный PDF в новой вкладке. Для .doc/.docx — карточка с иконкой документа и именем файла.

---

## Вариант 2 — iframe / object (встроенный просмотр)

**Идея:** Показать PDF в `<iframe src={fileUrl}>` или `<object data={fileUrl} type="application/pdf">` в уменьшенном блоке.

- **Плюсы:** Без библиотек, мало кода.
- **Минусы:** Зависит от браузера (не везде даёт красивую миниатюру), с cross-origin (Blob/другой домен) могут быть ограничения или пустой iframe.

---

## Вариант 3 — Только карточки с иконкой (без превью содержимого)

**Идея:** Вместо списка — сетка карточек как у Images: иконка PDF / документа, имя файла, размер, кнопка «Открыть» и «Удалить». Содержимое не показываем.

- **Плюсы:** Быстро, без зависимостей, единый стиль с блоком Images.
- **Минусы:** Не видно, что внутри документа — только тип файла и название.

---

## Вариант 4 — Серверное превью при загрузке

**Идея:** При загрузке PDF на бэкенде (Node) генерировать картинку первой страницы (например через `pdf-poppler`, `graphicsmagick` или облачный API), сохранять её в Blob/папку и отдавать URL превью. В админке показываем эту картинку как миниатюру.

- **Плюсы:** Один раз посчитали — быстрое отображение, можно одинаково для PDF и после конвертации .doc → PDF.
- **Минусы:** Нужны доработки API и хранилища, зависимость от инструмента на сервере.

---

## Рекомендация

- **Сделать быстро и стабильно:** Вариант 1 (PDF.js) для PDF + Вариант 3 (карточки с иконкой) для .doc/.docx.
- **Минимум кода:** Вариант 3 для всех типов (единые карточки, без превью содержимого).
- **Максимум качества и гибкости:** Вариант 4 (серверные превью), если готовы менять бэкенд.

---

## Имена + превью первой страницы (маленькая версия)

**Что даёт «видно содержимое первой страницы»:**

| Вариант | Превью 1-й страницы | Доп. нагрузка на сервер | Доп. хранилище / БД |
|--------|----------------------|--------------------------|----------------------|
| **1. PDF.js** | ✅ Да — рисуем в браузере | ❌ Нет | ❌ Нет |
| **2. iframe** | ⚠️ Иногда (зависит от браузера/URL) | ❌ Нет | ❌ Нет |
| **3. Только иконка** | ❌ Нет | ❌ Нет | ❌ Нет |
| **4. Серверное превью** | ✅ Да — картинка с сервера | ✅ Да (генерация при загрузке) | ✅ Да (файлы превью в Blob/хранилище) |

**Итог:**
- Нужно **превью первой страницы без затрат на сервер и хранилище** → **Вариант 1 (PDF.js)**. Всё в браузере: по ссылке на PDF рисуем первую страницу в маленький canvas, имя файла оставляем как есть.
- Вариант 4 добавляет нагрузку на сервер (генерация при каждой загрузке) и место в хранилище (файлы превью).

---

## Сравнение: серверный thumbnail vs клиентский PDF.js

| Критерий | **Серверный thumbnail** (Вариант 4) | **Клиентский PDF.js** (Вариант 1) |
|----------|--------------------------------------|-----------------------------------|
| **UX** | Лучше: показывается готовая картинка (PNG/WebP), быстро грузится. | Хуже: браузер каждый раз подкачивает весь PDF и рендерит первую страницу — тяжелее и для UX, и для трафика. |
| **Контроль** | Полный: размер, качество, формат превью задаёте на сервере. | Зависит от клиента: размер/качество в коде, но PDF всё равно качается целиком. |
| **Сервер** | Нагрузка при загрузке файла (pdfium/poppler → PNG/WebP). | Без доп. нагрузки: отдаёте тот же PDF. |
| **Хранилище** | Нужно хранить картинку превью рядом с PDF (Blob/файлы). | Ничего не храним — только исходный PDF. |
| **Трафик** | Меньше: клиент грузит маленькую картинку. | Больше: при открытии списка/карточки подгружаются целые PDF для рендера превью. |

**Рекомендация по совету:**

- **Лучший UX и контроль** → **серверный thumbnail** (генерация при загрузке, хранение PNG/WebP, в UI — картинка; по клику — открыть PDF). Подходит, если готовы добавить шаг в API при загрузке и место в хранилище.
- **Без изменений на сервере и без хранилища** → **клиентский PDF.js**. Подходит, если PDF немного и трафик/нагрузка на клиент допустимы.

**Итог:** Если можно добавить серверную генерацию и хранилище — **серверный thumbnail лучше**. Если нужно обойтись без бэкенда и хранилища — **PDF.js приемлемый компромисс**.

Напишите, какой вариант выбираете — реализую под него код в `FileUpload` и, при необходимости, в API.
