# План реализации: интерактивный план этажа (Floor Plan View)

Документ описывает план действий для реализации функционала **динамического интерактивного плана этажа**: загрузка плана (JPG/PNG), рисование зон квартир поверх изображения в системе, отображение статусов (продано / доступно / в резерве), подсветка при наведении, popup с данными и переход на страницу квартиры по клику.

---

## 1. Цели и результат

- **Для клиента:** Один блок в системе: загрузил план этажа (JPG/PNG) → в этой же системе нарисовал зоны по квартирам → на фронте план интерактивный (цвет по статусу, hover, клик в карточку квартиры).
- **Без внешних инструментов:** Всё делается в CRM; компании не обязаны использовать Figma или другие редакторы.
- **Формат планов:** Поддержка обычных растровых изображений (JPG, PNG); вектор не обязателен.

---

## 2. Текущее состояние (на момент составления плана)

- **BuildingFloorPlan:** одна картинка на этаж (`buildingId`, `floor`, `fileUrl`, `fileName`). Размеры изображения не хранятся.
- **Floor view** в `ApartmentsList`: показывается изображение плана и под ним список квартир этажа карточками; клик по карточке → `/apartments/[id]`.
- **Квартиры:** `status` (UPCOMING, AVAILABLE, RESERVED, SOLD), `floor`, `apartmentNo`, `sqm` и др. Отдельных полей «комнаты / санузлы / балконы» в схеме нет.
- **Привязки квартир к областям на плане:** нет; нужно добавить модель и редактор.

---

## 3. Модель данных

### 3.1. Размеры изображения плана

**Вариант A (рекомендуется):** добавить в `BuildingFloorPlan` поля:

- `imageWidth` (Int, nullable) — ширина изображения в пикселях при загрузке/сохранении.
- `imageHeight` (Int, nullable) — высота изображения в пикселях.

Размеры нужны для корректного пересчёта координат зон при отображении (например, при object-fit: contain). Можно выставлять при первой загрузке или при первом открытии редактора зон (получить размеры по `fileUrl` на бэкенде или в браузере и сохранить).

**Вариант B:** хранить размеры в отдельной таблице или в JSON зон. Менее удобно при изменении картинки плана.

### 3.2. Зоны на плане этажа

Новая сущность (например, `FloorPlanZone` или `BuildingFloorPlanZone`):

| Поле         | Тип     | Описание |
|--------------|---------|----------|
| id           | Int, PK | — |
| buildingId   | Int, FK → Building | — |
| floor        | Int     | Номер этажа |
| apartmentId  | Int, FK → Apartment | Квартира, которой соответствует зона |
| points       | Json    | Массив точек полигона (см. ниже) |
| createdAt    | DateTime | — |
| updatedAt    | DateTime | — |

**Формат `points` (нормализованные координаты 0–1):**

- Вариант 1: массив пар `[{ "x": 0.2, "y": 0.3 }, ...]` — доли от ширины и высоты изображения. При отрисовке: `x_px = x * imageWidth`, `y_px = y * imageHeight` (в координатах картинки), затем масштабирование под отображаемый размер.
- Вариант 2: массив пар в пикселях исходной картинки `[{ "x": 100, "y": 200 }, ...]` + обязательное хранение `imageWidth`/`imageHeight` у плана; при отрисовке пересчитывать в текущий размер отображения.

Рекомендация: **нормализованные 0–1** — проще при смене разрешения картинки и при разном масштабе отображения.

Ограничения:

- Уникальность: одна квартира на одном этаже — одна зона (или разрешить несколько зон на квартиру, если планировка разрывная; тогда при hover/click объединять). Для простоты: `@@unique([buildingId, floor, apartmentId])` или без unique, если решите несколько контуров на квартиру.

Индексы: `@@index([buildingId, floor])` для быстрой выборки зон этажа.

## 4. Миграции БД

1. Добавить в `BuildingFloorPlan` (если выбран вариант 3.1):
   - `imageWidth Int?`
   - `imageHeight Int?`
2. Создать таблицу зон (например `floor_plan_zones`) с полями из п. 3.2.
3. При удалении квартиры или плана этажа — удалять связанные зоны (onDelete: Cascade со стороны Building/Floor или явное удаление в сервисе).

---

## 5. API

### 5.1. Зоны плана этажа

- **GET** `/api/buildings/[id]/floor-plans/[floor]/zones`  
  Возврат: список зон этажа с полями `id`, `apartmentId`, `points`, при необходимости — данные квартиры (`apartmentNo`, `status`, `sqm`) для отрисовки и popup.  
  Права: авторизованный пользователь (или по текущей логике чтения планов).

- **PUT** или **POST** `/api/buildings/[id]/floor-plans/[floor]/zones`  
  Тело: `{ zones: [ { apartmentId, points: [{ x, y }, ...] }, ... ] }` — полная замена зон этажа или по одной зоне.  
  Права: только админ (аналогично загрузке/удалению плана).

- **DELETE** `/api/buildings/[id]/floor-plans/[floor]/zones/[zoneId]` (опционально) — удаление одной зоны.

При сохранении зон проверять: квартира принадлежит этому зданию и этажу; `points` — массив с минимум 3 точками.

### 5.2. Размеры изображения плана

- При **POST** загрузке плана (`/api/buildings/[id]/floor-plans`) опционально принимать `imageWidth`, `imageHeight` (если вычисляются на клиенте при выборе файла).
- Либо отдельный **PATCH** `/api/buildings/[id]/floor-plans/[floor]` с телом `{ imageWidth, imageHeight }` для сохранения размеров (например после первого открытия редактора).

## 6. Админка: редактор зон (рисование полигонов)

### 6.1. Место в UI

- В карточке/странице здания (Building edit), в блоке «Floor plans»: для каждого этажа, у которого есть загруженный план (JPG/PNG), добавить кнопку **«Редактировать зоны»** / «Настроить зоны квартир».
- По клику открывается экран редактора зон для выбранного этажа (отдельная страница или модальное окно на весь экран).

### 6.2. Редактор зон — сценарий

1. Показать изображение плана этажа (тот же `fileUrl`). Если у плана нет `imageWidth`/`imageHeight` — загрузить изображение, вычислить размеры, отправить PATCH и сохранить в состоянии.
2. Загрузить список квартир этого этажа (по `buildingId` + `floor`) и список уже сохранённых зон (GET zones).
3. Режимы:
   - **Добавить зону:** пользователь выбирает квартиру из выпадающего списка (или «сначала нарисовать, потом привязать»). На изображении кликами ставит точки полигона. Замыкание контура: двойной клик или кнопка «Закрыть полигон». После закрытия — сохранить зону (добавить в список и отправить на сервер или накапливать и сохранять всё кнопкой «Сохранить»).
   - **Редактировать зону:** выбор существующей зоны (по списку или клик по полигону) → отображение точек-вершин, перетаскивание вершин для изменения формы, опционально добавление/удаление точек.
   - **Удалить зону:** выбор зоны → кнопка «Удалить».
4. Визуализация в редакторе: полигоны можно рисовать полупрозрачным цветом, разным по квартирам; номера квартир подписать на полигоне или в списке рядом.
5. Кнопки «Сохранить», «Отмена» (выйти без сохранения).

### 6.3. Техническая реализация редактора

- **Отрисовка:** поверх `<img>` плана — элемент с тем же aspect ratio и абсолютным позиционированием (или SVG viewBox под размер картинки). Координаты кликов переводить в нормализованные (0–1) относительно размеров отображаемой картинки (учесть object-fit и размер контейнера).
- **Рисование полигонов:** при клике добавлять точку в массив; отрисовывать текущий полигон (незамкнутый — линией, замкнутый — заливкой). Двойной клик или кнопка — закрыть контур (дублировать первую точку в конец или просто считать замкнутым при отрисовке).
- Библиотеки (на выбор): без библиотек (чистый React + SVG/Canvas), или Fabric.js, Konva (react-konva), или специализированные для image annotation (например, react-image-annotate и аналоги). Для минимальной зависимости достаточно SVG с `polygon`/`path` и обработкой мыши.

## 7. Публичный вид: интерактивный план этажа

### 7.1. Где показывать

- В текущем **Floor view** в `ApartmentsList`: вместо (или вместе с) статичной картинки и списка карточек — компонент «Интерактивный план этажа».
- Условие: для данного этажа есть план и есть хотя бы одна зона; иначе показывать текущее поведение (картинка + список карточек).

### 7.2. Поведение компонента

- Отображение: изображение плана + поверх SVG (или Canvas) с полигонами по сохранённым зонам.
- Масштаб: координаты из БД (0–1) пересчитывать в текущие размеры отображаемой картинки (учесть object-fit: contain и размер контейнера, чтобы overlay совпадал с планом).
- Цвет заливки/обводки по статусу квартиры, например:
  - SOLD — серый, полупрозрачный; подпись «Продано» (или иконка).
  - RESERVED — один цвет (например жёлтый/оранжевый).
  - AVAILABLE — другой (например зелёный или нейтральный).
  - UPCOMING — нейтральный.
- Hover: подсветка соответствующей квартиры (другой цвет/обводка), курсор pointer.
- Click: переход на страницу квартиры `router.push(\`/apartments/${apartmentId}\`)`.
- Popup при hover (или при клике до перехода): блок с номером квартиры, площадью (sqm), при необходимости — кнопка «Подробнее» (тот же переход). При отсутствии полей «комнаты/санузлы/балконы» в БД — не показывать их; при добавлении полей в будущем — вывести в popup.

### 7.3. Реализация компонента

- Имя, например: `InteractiveFloorPlan`.
- Props: `floorPlanUrl`, `imageWidth`, `imageHeight`, `zones` (массив с `apartmentId`, `points`, и данными квартиры для цвета и popup), `onApartmentClick(apartmentId)`.
- Отрисовка: контейнер с `<img>` и поверх — `<svg>` с теми же пропорциями; для каждой зоны — `<path>` или `<polygon>` из `points`; обработчики `onMouseEnter`, `onMouseLeave`, `onClick`. Popup — абсолютно позиционированный div рядом с курсором или над зоной.
- Доступность: по возможности клавиатурный фокус и aria-метки для зон (квартира N, статус, переход по Enter).

## 8. Сервисы и кэш

- **Building service:** при запросе здания с планами при необходимости подгружать зоны по этажам (или отдельным запросом зон для выбранного этажа в редакторе и во view).
- **Зоны:** отдельный сервис или методы в building service: `getZones(buildingId, floor)`, `saveZones(buildingId, floor, zones)`.
- Кэш: при изменении зон инвалидировать кэш зданий/планов так же, как при загрузке или удалении плана (см. текущие `invalidateCache` в floor-plans API).

## 9. Этапы и приоритеты

| Этап | Задачи | Оценка |
|------|--------|--------|
| 1. Модель и API | Prisma: BuildingFloorPlan (imageWidth, imageHeight), таблица зон; миграция; API GET/PUT zones, опционально PATCH размеров плана | 0.5–1 дн |
| 2. Редактор зон в админке | Страница/мода «Редактировать зоны»; загрузка плана и списка квартир; рисование полигонов (клики, закрытие, привязка к квартире); сохранение через API | 2–3 дн |
| 3. Интерактивный вид | Компонент InteractiveFloorPlan; интеграция в Floor view; цвета по статусу, hover, click, popup | 1.5–2 дн |
| 4. Доработки | Редактирование/перемещение вершин существующих зон; удаление зоны; расчёт/сохранение imageWidth/Height при загрузке; доработка popup (при появлении полей комнаты/санузлы/балконы) | 1–2 дн |

**Минимум для «работающего» сценария:** этапы 1–3 (ориентировочно 4–6 рабочих дней).  
**С удобным редактором (перетаскивание вершин и т.п.):** + этап 4 (итого около 7–10 дней).

## 10. Риски и упрощения

- **Смена изображения плана:** при замене картинки этажа координаты зон (0–1) останутся корректными, если пропорции не меняются; при смене пропорций зоны могут «съехать». Вариант: при замене картинки сбрасывать зоны с предупреждением или пересчитывать точки по новым размерам (если хранятся в пикселях).
- **Упрощение редактора:** первый релиз — только «добавить полигон по кликам» и «удалить зону»; редактирование вершин — во втором заходе.
- **Производительность:** при большом числе квартир на этаже — один SVG с множеством path; при необходимости можно разбить на несколько слоёв или виртуализировать подсказки.
